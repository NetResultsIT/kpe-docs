Detailed Description

This is the main class of the KPE to use with the Qt C++ libraries. KpeQt is the main object that includes all the business logic of a "Kalliope-based" softphone written with Qt To test it you need just to :

  • set the login data (username, password, pbxaddress) via setLoginData()
  • Initialize KPE with the init() method
  • Connect to PBX via connectToKalliope() The KpeQt will automatically perform all necessary operation to connect to kalliope and provision the engine to be ready to be used
Note
The KpeQt is studied to be used within applications written with the Qt C++ libraries.
All the events generated by KPE during its use are handled by Qt Signal/Slot mechanism, refer to documentation below to check out all the available signals.

◆ setSipProvisioningData()

int KpeQt::setSipProvisioningData ( const QString & jsonSipProvData)

Set Sip provisioning data.

Parameters
jsonSipProvDataA string representing a Json object that contains provisioning data
Returns
KPEErrorNoError or a negative error code
Version
1.26.0
Note
This method can only be called before init()

The param is expected to be a Json object containing at least the following key/value pairs:

  • sipUsername and sipPassword (strings): the Sip account credentials;
  • sipRegistrarIp (string) and sipRegistrarPort (int): the Sip registrar address;
  • sipTransportProtocol (string): one of UDP, TCP, TLS, WS, WSS;
  • sipMediaProtocol (string): one of RTP, SRTP;
  • codecs (array of strings): ordered codec list; possible values are opus, alaw, ulaw, g729, gsm;
  • dtmfMode (string): one of SIPINFO, RFC2833;

The following keys are optional and, if not present, related values will be set to their default ones:

  • sipProxyIp (string) and sipProxyPort (int): Sip proxy address (default: equal to registrar);
  • sipTos and audioTos (int): QoS values (default, respectively: 24 and 46).
  • overrideKalliopeProvisioningData (bool): whether to keep this custom provisioning data even though provisioning data received from PBX (upon successful connection to Kalliope) is different (default: false). Further details below.

This method is useful when Sip provisioning data is known in advance, i.e. before such data is received from Kalliope on successful login. Data set through this method will speed up Voip engine initialization and provisioning, since such operations will be performed during the init() execution, removing the need to wait for a successful login to Kalliope (see connectToKalliope()).

The choice to override Kalliope provisioning data is licit only in the following cases:

  • the given Sip account must exist on PBX (obviously, otherwise the register operation will fail);
  • the given Sip account must be of a compatible type (i.e. either KPE Desktop or KPE Mobile, according to the current app platform);
  • the given Sip account must be associated to this KPE client, PBX-side.

If the above conditions are met, then the other params can be overridden (registrar and proxy address, transport and media protocol, codecs, DTMF mode, QoS values). Otherwise, Voip engine will be forcibly shut down and re-provisioned with Kalliope data. User attempting to override Kalliope data must be aware that such an "early" Sip provisioning might fail due to incorrect data or data that is not compatible with the general Sip configuration PBX-side (e.g. transport and/or media protocol not enabled PBX-side).

If user decides not to override data received from Kalliope, then KPE will automatically update Sip provisioning data, after Kalliope data is received upon successful login. If needed, Voip engine will be restarted as soon as there are no more active calls. In the event that the data received from the server lacks certain parameters (e.g. transport/media protocols, codecs, DTMF mode, QoS values), the user-defined values for those parameters will be preserved. In case some provisioning data has changed, but no Voip engine restart was necessary, the sigSipProvisioningCompleted() signal will be emitted anyway in order to inform user that current Sip configuration has changed.

In every moment, the current Sip provisioning data can be retrieved by calling getCurrentSipProvisioningData(), which may be useful to store app-side for future calls to this method (i.e. when KPE is newly allocated)

    • msg: a string representing the network error reported by socket

  • phoneAccountCodecs: it is only set if phone mode is in operation. If set to true, indicates that Sip codecs list related to currently used phone Sip account has been changed.
