PBX_DISCONNECTION_EVENT 

A CTI connection to PBX was interrupted. Event details:

{
"reason": "...",
"details": {...}
}

where

  • reason is a string that contains a human-readable reason of the disconnection;
  • details is an (optional) object containing a <key, value> map of reason details, which may have any name and type according to reason specifications.

The following is a list of all possible disconnection reasons and possible related details:

  • Unknown: unknown reason. Details:
    • msg: if existing, a string that contains a message that may help in identifying the unknown reason
  • ServerCertificateNotKalliope: the SSL certificate exposed by server is not signed by Kalliope CA. By default, connection is not allowed if certificate is not Kalliope-signed. A possible way to avoid this error is to add server certificate to whitelist (see addServerCertificateToWhitelist method). Details:
    • cert: a string containing the server certificate in PEM format;
    • valid: a boolean indicating whether certificate is valid or not (although not signed by Kalliope CA);
    • sslErrors: an array of strings, each one containing an SSL error reported by socket during handshake. It is empty in case valid is true
  • UnknownServerType: server is not a Kalliope CTI server
  • UnsupportedServerApi: the API exposed by Kalliope CTI server are not supported (i.e. too old Kalliope firmware)
  • RedirectionLoop: Kalliope CTI server attempted to redirect connection to the same IP/port
  • AuthFailed: user authentication to Kalliope CTI server failed
  • NoLicenseFound: no valid license was found for user requesting either phone or remote control capabilities
  • NoSipAccountAvailable: (Kalliope fw version < 4.15.4) no available Sip account was found for user requesting phone capabilities
  • NoRemotePhone: (Kalliope fw version < 4.15.4) no remotely-controllable deskphone was found for user requesting remote control capabilities
  • AdminRequest: admin reason, e.g. server restart or programmed server updates occurring
  • UserRequest: disconnection requested by user
  • NotAuthorized: login attempted with valid but not authorized credentials (e.g. Kalliope tenant/machine admin)
  • NoCapability: Kalliope CTI server granted no capabilities to logged user
  • NoExten: Kalliope CTI server found no exten related to logged user
  • WrongAppMode: Kalliope CTI server granted wrong app mode (i.e. desktop in place of mobile environment or vice versa)
  • ConnectionTimeout: Kalliope CTI server did not answer keepalive messages for a while
  • NetworkError: TCP error. Details:
    • msg: a string representing the network error reported by socket
  • InternalServerError: internal error occurred in Kalliope CTI server (failed to perform necessary internal operations)

SIPACCOUNTLIST_UPDATED 

Note: only emitted if connected to a Kalliope with fw >= 4.15.4. SIP account list updated. Event details:

{
"remoteControlList": true/false,
"phoneAvailableList": true/false,
"phoneModeCompromised": true/false,
"phoneAccountReleased": true/false,
"phoneAccountCodecs": true/false,
"phoneAccountTos": true/false,
"phoneAccountDtmfMode": true/false
}

where:

  • remoteControlList: if set to true, indicates that the list of Sip accounts used by remotely-controllable devices has changed;
  • phoneAvailableList: if set to true, indicates that the list of free (i.e. not associated to any KPE client) accounts suitable for phone mode has changed;
  • phoneModeCompromised: it is only set if phone mode was in operation (i.e. SIPPROVISIONING_SUCCESSFUL event was already emitted). If set to true, indicates that phone mode is compromised and Voip engine will be soon forcibly turned off. Once done, VOIPENGINE_SHUTDOWN event is emitted;
  • phoneAccountReleased: it is only set if phone mode was in operation and phoneModeCompromised is also set. If set to true, indicates that someone released the currently used phone Sip account on the PBX, i.e. the association between this client and its current phone Sip account has been explicitly removed Kalliope-side. In this case, also phoneAvailableList flag will be set, because the released account has now become available (for further details see VOIPENGINE_SHUTDOWN event);
  • phoneAccountPassword: it is only set if phone mode is in operation. If set to true, indicates that password of currently used phone Sip account has been changed. Upon this event emission, Voip engine will already be up-to-date, active calls will not be aborted and Voip engine does not need to be re-provisioned (i.e. phone mode is not compromised);
  • phoneAccountTos: it is only set if phone mode is in operation. If set to true, indicates that Sip TOS values for the currently used phone Sip account have been changed. Upon this event emission, Voip engine will be already up-to-date, active calls will not be aborted and Voip engine will not be turned off (i.e. phone mode is not compromised). Anyway, new TOS values won't affect currently active calls nor future calls, until a new provisioning is done. To force a new provisioning of Voip engine, a new login is necessary, otherwise KPE must be explicitly shutdown and re-initialized;
  • phoneAccountCodecs: it is only set if phone mode is in operation. If set to true, indicates that Sip codecs list related to currently used phone Sip account has been changed. Upon this signal emission, Voip engine will already be up-to-date, active calls will not be aborted and Voip engine does not need to be re-provisioned (i.e. phone mode is not compromised);
  • phoneAccountDtmfMode: it is only set if phone mode is in operation. If set to true, indicates that DTMF mode related to currently used phone Sip account has been changed. Upon this signal emission, Voip engine will already be up-to-date, active calls will not be aborted and Voip engine does not need to be re-provisioned (i.e. phone mode is not compromised);
  • otherList: if set to true, indicates that there are changes in the list of Sip accounts that have no impact on phone mode nor on remote-control mode ("other" accounts).

To obtain the updated list (or sublist) of Sip accounts, use KpeQt::getSipAccountList().

SIPPROVISIONING_SUCCESSFUL

SIPPROVISIONING_FAILED 

SIP provisioning of voip engine failed. This is also emitted in case no free and compatible phone Sip account is currently associated to this client, when logged in phone mode. Use getSipAccountList() to find if a phone Sip account is currently provisioned to KPE. If not, it means that no associated phone Sip account has been found by Kalliope for this client at login. From version 1.4.0, if connected to a Kalliope with fw version >= 4.15.4, it is possible to recover from this: see event SIPACCOUNTLIST_UPDATED event and KpeQt::getSipAccountList() to find if a free and compatible phone Sip account appears and request its explicit association to this client through KpeQt::requestPhoneSipAccountAssociation(). If a non-empty phone Sip account is currently provisioned to KPE instead, then the issue is not recoverable (check logs). Also note that this signal is emitted every time Sip provisioning fails, i.e. also after a new phone Sip account association has been done. In fact, after Kalliope notifies that a new phone Sip account has been successfully associated to this client, KPE attempts to provision Voip engine

VOIPENGINE_SHUTDOWN 

Note: only emitted if connected to a Kalliope with fw >= 4.15.4. Event details:

{
"phoneAccountExplicitlyReleased": true/false
}

Voip engine has been forcibly shutdown. The cause is that the Sip account provisioned to Voip engine has been unrecoverably changed and its association to this KPE client has been removed Kalliope-side (possible reasons: name or type were changed; it was removed; it was explicitly released, i.e. its association to this KPE client was removed following an explicit request). In this case, phone mode is compromised and no more in operation and Voip engine is turned off. Moreover, before this event is emitted, every active call is automatically aborted. This fact should be taken into account, because emission of CALL_TERMINATED events is not guaranteed in this case. To put Voip engine back in operation, it is necessary that a new association to a free and compatible KPE account is requested to Kalliope: use KpeQt::requestPhoneSipAccountAssociation(), once an available Sip account (if any) has been chosen (see KpeQt::getSipAccountList()). Another possibility is to let Kalliope do the job: if client disconnects and reconnects again, Kalliope automatically associates the client to an available Sip account, if any. The aim of phoneAccountExplicitlyReleased flag is to let KPE user make an informed choice: consider that account has been released for some reason (likely, to let another client use it), before deciding to immediately request again an association to this Sip account
